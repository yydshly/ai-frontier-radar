关于系统

AI Frontier Radar 如何从精选来源生成可信的中文 InsightCard。

🧭 系统定位

AI Frontier Radar 不是：

  • 普通 RSS 阅读器 — 它不只是订阅和推送
  • 普通翻译器 — 它不是逐句翻译，而是结构化洞察编译
  • 全网爬虫 — 它只从精选来源获取，不追求全面覆盖

AI Frontier Radar 是：

面向 AI 从业者和独立开发者的前沿资料中文编译工作台。

它的核心价值是：帮助你在海量英文资料中，快速筛选高价值内容，生成结构化的中文洞察，并形成可执行的个人判断。

V1.0-alpha 当前优先验证：本地个人工作台 + 单篇资料编译链路的完整性。

不追求来源数量和覆盖率，优先确保每一条线索都可信、可追溯、可执行。

🔄 整体处理链路

AI Frontier Radar 的完整处理链路如下：

1精选来源
→
2来源探测
→
3候选资料发现
→
4URL 类型识别
→
5正文抓取与清洗
→
6去重
→
7LLM 结构化分析
→
8中文 InsightCard
→
9中英双语核心理解
→
10用户判断
→
11Markdown 报告 / 行动任务导出

⚠️ 重要：不是所有 URL 都会直接进入总结。

系统会先判断 URL 类型（单篇文章 / 列表页 / 分页页 / Feed），再决定是否允许编译。这避免了把"发现入口"误当"内容来源"处理。

📡 前沿来源如何控制

当前来源管理方式：config/sources.example.yaml + 数据库同步。

每个来源的核心配置：

字段说明
source_key来源唯一标识（如 openai_news）
source_type类型：rss / html_index / manual
fetch_strategy抓取策略：rss / html_index / manual
enabled是否启用
category分类：company / research / paper 等
tags标签，用于相关性判断
relevance_hint相关性提示词，影响 LLM 判断
fetch_interval_hours探测频率

产品原则：优先精选高价值来源，而不是全网乱爬。

V1.0-alpha 精选来源：OpenAI / Anthropic / Google DeepMind / Hugging Face / Stanford HAI / MIT News AI / NVIDIA AI / Microsoft AI / Mistral / Cohere 等。

来源数据如何获取

信息来源页通过现有 SourceFetchService 运行探测，按来源配置选择 RSS 或 HTML index 策略。

每次探测都会生成 FetchRun，记录发现数量、新增数量、更新数量和失败数量；单条候选资料进入 SourceItem / 候选池，后续再由用户加入生成队列。

🏷️ URL / 文档类型识别与策略路由

用户输入 URL 后，系统先判断它是什么类型，再决定怎么处理。

类型判断由 app/intake/url_classifier.py 的规则分类器完成，不需要 LLM。

类型示例策略说明
article /discover/blog/sima-2-agent/ ✅ 允许编译 单篇博客 / 新闻 / 研究文章
pdf /report.pdf ✅ 允许编译 PDF 报告
listing /blog/ 🚫 只用于发现 列表页，不是单篇文章
pagination https://deepmind.google/blog/page/3/ 🚫 阻止编译 分页页，不是内容页
tag_or_category ?tag=agents 🚫 只用于发现 标签页 / 分类页，不是内容
unknown 无法判断 ⚠️ 谨慎尝试 无法判断时记录日志，等待人工确认

遇到阻塞和失败怎么办

列表页、分页页、Feed 等入口不会直接进入总结；系统会创建可解释的失败或阻塞记录，帮助用户知道问题出在哪里。

生成失败的候选资料可以在候选池、运行详情或生成队列里重试，所有生成动作保持 POST 提交。

🧠 LLM 分析和输出边界

中文摘要 vs 中英双语核心理解：

中文摘要中英双语核心理解
用途 快速看懂资料 深度阅读和长期沉淀
语言 中文 英文原文 + 中文解说
内容 摘要 + 关键事实 + 技术洞察 英文核心观点 + 原文主张 + 关键词汇 + 中文解说 + 保真提示

V1.0-alpha 不做什么：

  • 不做多语言混合输出（始终保留英文原文作为事实锚点）
  • 不做实时信息追踪（无后台任务、无定时抓取）
  • 不做全网搜索（只从精选来源获取）
  • 不追求 LLM 输出的"完整性"（允许 JSON 解析失败降级）

关键技术选型

当前应用保持 FastAPI + Jinja2 + SQLite + 纯 HTML/CSS 的轻量形态，不引入前端框架或新的构建链路。

URL 安全由 safe_external_url 做 http/https 白名单过滤；来源探测、候选池和生成队列继续复用现有服务。
