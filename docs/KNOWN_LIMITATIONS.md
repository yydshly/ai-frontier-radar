# V1.0-beta Known Limitations

## 当前定位

V1.0-beta 是**第一可用闭环版本**，验证「信息来源 → 探测 → 候选池 → 加入生成 → 生成队列 → InsightCard」完整链路可在本地跑通。

核心目标是让用户能在本地完成一个完整的资料发现到洞察生成周期。

---

## 已知限制

### 1. BackgroundTasks 不是持久队列

**当前使用 FastAPI 的 BackgroundTasks 机制。**

**影响：**
- 服务重启后，所有 running 状态的任务和已加入生成队列但未完成的任务会丢失
- 任务不会自动恢复或重试

**当前 workaround：** 手动重新点击「运行探测」。

**建议（未来版本）：** 引入持久任务队列（如 Celery + Redis、BullMQ 或类似方案）。

---

### 2. 10 分钟 running 去重不是分布式锁

**当前去重逻辑：同一来源 10 分钟内有 running 状态的 FetchRun 则不再创建新任务。**

**影响：**
- 单进程部署：去重生效
- 多进程部署（如 gunicorn -w N）：各进程内存不共享，可能同一来源并发运行
- 不是真正的分布式锁，不能保证跨进程/跨机器的互斥

**当前 workaround：** 使用单进程部署（uvicorn 默认）可避免此问题。

---

### 3. delta new / seen / updated 仍是时间窗口估算

**FetchRun 详情页的「新增 / 已见 / 已更新」统计是基于时间窗口的估算，不是精确的变更检测。**

**影响：**
- 同一篇文章可能在不同探测周期内重复出现
- 已存在的条目不会重新抓取正文内容

**建议（未来版本）：** 基于 content hash 的精确去重 + 增量更新。

---

### 4. HTML index 仅抓前 15 个候选详情页 metadata

**HTML Index 来源探测时，每个来源最多抓取 15 个详情页的 metadata（og:title、description 等）。**

`MAX_DETAIL_FETCHES_PER_SOURCE = 15`

**影响：**
- 大型博客/新闻列表页超过 15 篇时，后面的内容不会抓取详情 metadata
- 标题降级为列表页链接文本，可能出现「Learn More」等弱标题

**当前 workaround：** 弱标题在候选池和生成队列中显示为「标题待修复」，不影响生成链路。

---

### 5. 旧脏标题需要 repair 脚本或重新探测修复

**弱标题（「Learn More」「FEATURED」等）只在当前探测周期内通过详情页 metadata 尝试修复。**

**影响：**
- 历史已有脏标题的 SourceItem 不会自动修复
- 需要重新运行探测或使用 repair 脚本

**建议（未来版本）：** 引入批量 repair 脚本或脏标题自动修复服务。

---

### 6. 中文摘要主要依赖英文 metadata

**当前中文摘要仍主要来自英文 metadata 的 description 和摘要字段。**

**影响：**
- 尚未完成 AI 中文一句话摘要生成
- 英文摘要中可能存在不完整或有歧义的内容

**建议（未来版本）：** 引入 AI 中文一句话摘要生成模块。

---

### 7. 真实来源全量验证尚未完成

**部分已配置来源尚未完成真实探测全量验收。**

**影响：**
- 部分来源可能在实际探测中失败
- 探测稳定性尚未完全验证

**建议（未来版本）：** 完成全量真实来源覆盖验收。

---

### 8. 新增来源偏配置文件 / 数据库层

**当前新增来源仍需通过 YAML 配置文件或数据库直接操作。**

**影响：**
- 网页端新增来源入口尚未完成
- 用户添加新来源不够直观

**建议（未来版本）：** 完成网页新增来源表单入口。

---

### 9. SQLite 本地数据库

**当前使用 SQLite（文件数据库）。**

**影响：**
- 适合本地演示和开发
- 不适合高并发（多人同时使用）
- 不适合大数据量（ > 10 万条记录）

**建议（未来版本）：** 迁移到 PostgreSQL。

---

### 10. 无数据库迁移系统

**当前使用 `Base.metadata.create_all()` 初始化数据库。**

**影响：**
- 模型（表结构）变更后，旧的 SQLite 文件不能自动迁移
- 需要手动删除旧 DB 或重建

**当前 workaround：** 开发阶段删除 `data/` 下的 `.db` 文件重建。

---

### 11. 真实网页抓取不稳定

**部分来源可能遇到：**
- HTTP 403 / 429（来源站点反爬）
- 页面结构变化（CSS 选择器失效）
- RSS 返回空数据
- 候选链接质量参差不齐（过滤逻辑仍有误判可能）

**影响：** 某些来源可能在首次探测时失败，需要重试或人工介入。

---

### 12. 真实 LLM 输出质量依赖人工判断

**mock 验收只验证链路通不通。**

**真实 LLM 输出仍需要：**
- 保真检查（英文摘要是否忠实原文）
- 中文解释检查（是否曲解原意）
- 用户最终人工判断

**当前产品中：** 没有自动评估 LLM 输出质量的功能，纯靠人工判断。

---

### 10. /cards 和首页部分统计存在分页 / 性能优化需求

**当前 `/cards` 和首页统计查询尚未实现分页。**

**影响：**
- 数据量增长后，页面加载可能变慢
- 需要后续引入分页机制

**建议（未来版本）：** 引入分页或虚拟滚动。

---

### 13. 语音播报尚未实现

**语音播报（TTS）作为后续方向预留，尚未落地实现。**

**建议（未来版本）：** 完成中文一句话摘要后，可接入 TTS 模块实现播报。

---

### 14. quick_test / smoke_test / acceptance 定位不同

| 脚本 | 定位 | 何时跑 |
|------|------|--------|
| `quick_test.py` | 开发中快速自检 | 每次 commit 前 |
| `smoke_test.py` | PR-ready 完整回归 | PR 前 |
| `acceptance_demo_*.py` | 完整用户链路验收 | 大版本发布前 |

三者不能互相替代。

---

## 不在 V1.0-beta 范围内的能力

以下能力在 V1.0-beta 中**刻意不做**，属于未来版本考虑：

| 能力 | 原因 |
|------|------|
| 持久任务队列 | 先验证链路，再考虑后台任务稳定性 |
| 多用户/登录注册 | MVP 阶段聚焦单用户主链路 |
| 分布式部署 | 单进程部署足够验证产品假设 |
| 向量数据库/语义搜索 | 需要先有真实数据积累 |
| 知识图谱 | 需要先有实体和关系数据 |
| 自动周报推送 | 需要先有真实消费数据 |

---

## 下一阶段建议

详细优先级规划见 [docs/NEXT_EXECUTION_PLAN.md](docs/NEXT_EXECUTION_PLAN.md)。

1. **引入持久任务队列** — BackgroundTasks 换为 Celery/BullMQ 等
2. **脏标题 repair 脚本** — 批量修复历史弱标题
3. **数据库迁移策略** — 引入 Alembic 或类似工具
4. **分布式去重锁** — Redis 或数据库行锁实现真正的分布式互斥
5. **真实来源质量策略** — 改善 HTML 索引过滤、RSS 容错
6. **页面体验打磨** — 加载状态、错误提示、空状态 UI
