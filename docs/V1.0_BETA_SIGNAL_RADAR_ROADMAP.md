# V1.0-beta Signal Radar 路线图

**Date:** 2026-06-08
**Stage:** V1.0-alpha → V1.0-beta 方向设计

---

## 1. 当前阶段判断

V1.0-alpha 已完成"单篇资料编译器"的核心闭环：

```
SourceItem → InsightCard → 用户判断 → 完整报告 / 行动任务
```

V1.0-beta 的目标是将其演进为"高质量来源接入 + 候选池 + 轻量聚类 + 主题编译"的信号雷达。

---

## 2. 为什么不新建 v2 仓库

- 当前 `ai-frontier-radar` 已有完整的演示数据、CI/CD、本地健康检查和用户基础
- V1.0-beta 是产品模型升级，不是新项目，不需要重写所有历史
- 保留可运行的 alpha 底座，可以在不破坏现有流程的前提下并行开发 beta 功能
- GitHub repo 已经积累了 commits、issues、CI run，继续沿用

---

## 3. V1.0-beta 产品目标

**Signal Radar = 发现信号 + 聚类主题 + 沉淀洞察**

```
高频：发现 AI 前沿新动态（来源探测）
中频：主题级多源聚类编译（信号聚合）
低频：深度报告与行动追踪（洞察沉淀）
```

---

## 4. 当前 alpha 已有能力

| 能力 | 现状 |
|------|------|
| 精选来源配置（15个） | RSS / HTML Index 探测 |
| SourceItem 收件箱 | 发现 → 编译状态追踪 |
| 单篇 URL 编译 | POST URL → InsightCard |
| 中英双语核心理解 | English Core Summary + 中文解说 |
| 用户判断闭环 | 5 种判断 + 备注 |
| 完整报告导出 | Markdown 文件下载 |
| 首页工作台 | 统计 + 推荐流程引导 |
| 失败卡片清理 | 删除 failed card |

---

## 5. 当前 alpha 不足

| 问题 | 影响 |
|------|------|
| 来源接入靠手动配置 | 新来源需要代码修改 |
| SourceItem 是扁平的发现列表 | 无法按主题组织候选资料 |
| 每篇独立编译，无跨源聚类 | 同一主题的多篇报道无法联合分析 |
| 探测阶段就抓全文 | 带宽浪费，容易触发反爬 |
| 无超时 / 重试 / 配额控制 | 探测可能卡死或被封 |
| 无增量更新机制 | 重复内容无法识别和跳过 |
| URL 分类只做一次 | 无法跟踪来源结构变化 |

---

## 6. 高质量来源分层

V1.0-beta 将来源分为三层：

| 层级 | 来源类型 | 探测策略 |
|------|---------|---------|
| P0 实时 | 公司博客、论文 | 定期轮询，优先抓取元数据 |
| P1 低频 | 研究机构、年度报告 | 手动或定期 |
| P2 参考 | 社区讨论、工具发布 | 按需抓取 |

**每层来源配置：**

```python
SourceEntry:
  source_key: str          # 全局唯一标识
  name: str               # 显示名
  category: str            # company / research / paper / ...
  homepage_url: str       # 入口页（用于探测）
  feed_url: str | None    # RSS/Atom（优先）
  fetch_strategy: str      # rss / html_index / manual_pdf
  relevance_hint: str      # 领域关键词（用于过滤）
  priority: int            # P0/P1/P2
  enabled: bool
  fetch_interval_hours: int
```

---

## 7. 来源入口 SourceEntry 设计

用 `SourceEntry` 替代当前的 `Source` + `homepage_url` 硬编码模式。

```
SourceEntry（配置层）
  ↓ 每次探测生成
FetchRun（执行记录）
  ↓ 探测结果
CandidateItem（候选条目，元数据级）
  ↓ 用户选择 / 自动过滤
SourceItem（编译对象，完整 URL）
  ↓ 编译
InsightCard
```

**CandidateItem vs SourceItem 区别：**

| 字段 | CandidateItem | SourceItem |
|------|--------------|-----------|
| 完整正文 | 无（只存元数据） | 有 |
| 编译状态 | 无 | 有 |
| 用途 | 候选池筛选 | 实际编译 |
| 加载时机 | 探测时立即生成 | 用户点编译时 |

---

## 8. 候选池 Candidate Pool 设计

候选池是 V1.0-beta 的核心新概念：

```
候选池 = 已探测但尚未编译的所有 URL 元数据
         按来源 / 时间 / 关键词 / 相似度组织
         支持分页浏览
         支持批量选择编译
```

**核心操作：**

```python
# 候选池浏览
GET /candidate-pool
  ?source_key=deepmind_blog
  &status=discovered
  &since=2026-06-01
  &page=1

# 批量加入编译队列
POST /candidate-pool/batch-compile
  body: { candidate_ids: [1, 2, 3] }

# 批量标记为忽略
POST /candidate-pool/batch-ignore
  body: { candidate_ids: [4, 5] }
```

**候选池状态机：**

```
discovered → compiling → compiled
                   ↘ ignored
                   ↘ manual_required（PDF/长文需要用户确认）
```

## 9. 后续 SignalCluster 轻量聚类路线

SignalCluster 是主题级编译的核心：

```
SignalCluster:
  cluster_id: str
  topic: str                          # "OpenAI o3 Model Release"
  sources: list[str]                  # 参与的来源
  candidate_ids: list[int]            # 涉及的候选条目
  insight_card_ids: list[int]         # 生成的 InsightCards
  status: pending | compiling | completed | stale
  created_at: datetime
  updated_at: datetime
```

**触发条件（V2 阶段，不在 V1.0-beta 第一阶段落地）：**

- 同一主题在 24h 内有 N 篇新报道（N 可配置）
- 用户手动选中多个 SourceItem 发起聚类
- 来源首页出现"专题"标记

**聚类编译链路（V2 阶段）：**

```
SignalCluster
  → 多源内容抓取（带去重）
  → ClusterInsightCard（主题级洞察）
  → TrendRecord（趋势记录）
  → ActionTask（跨源行动建议）
```

## 10. 阶段规划

### V1.0-beta 第一阶段（本次文档范围）

**目标：高质量来源接入与候选池**

- [ ] `SourceEntry` 配置系统（替代硬编码 `homepage_url`）
- [ ] `CandidateItem` 模型（候选条目元数据）
- [ ] `FetchRun` 状态跟踪（start / running / success / partial_failed / failed）
- [ ] 候选池浏览页面（分页）
- [ ] 批量选择加入编译队列
- [ ] 批量忽略 / 标记
- [ ] 超时 / 重试 / `max_items` / `max_pages` 控制
- [ ] PDF 延迟解析（用户确认后才抓全文）
- [ ] 增量更新（基于 content_hash 去重）
- [ ] `SignalCluster` 数据模型（仅模型，不实现聚类逻辑）

**交付物：** 候选池 UI + FetchRun 状态追踪 + 来源配置化

### V2 阶段（后续）

- [ ] `SignalCluster` 聚类逻辑
- [ ] 主题级 `ClusterInsightCard`
- [ ] 趋势记录 `TrendRecord`
- [ ] 跨源行动任务 `ActionTask`
- [ ] 多源验证机制

---

## 11. 非目标

以下不在 V1.0-beta 范围内：

- 不做向量数据库 / embedding相似度搜索
- 不做自动翻译 pipeline
- 不做多语言支持
- 不做用户认证 / 多用户
- 不做 SaaS 部署
- 不做移动 App
- 不重构现有 alpha 流程（单篇编译继续工作）

---

## 12. 验收标准

### V1.0-beta 第一阶段验收

| 标准 | 验证方式 |
|------|---------|
| 新来源接入不需要代码修改 | 通过 SourceEntry 配置页添加新来源 |
| 候选池分页浏览正常 | GET /candidate-pool 返回分页数据 |
| 批量选择编译生成多张 InsightCard | POST /candidate-pool/batch-compile |
| PDF 不自动抓全文 | 候选池中 PDF 条目显示"需确认" |
| FetchRun 状态正确记录 | 查看 FetchRun 记录的状态和耗时 |
| 增量更新去重 | 重新探测不产生重复 SourceItem |
| 超时控制生效 | 模拟慢响应验证 timeout |
| 现有 smoke_test 全部通过 | python scripts/smoke_test.py |
| alpha 流程不受影响 | 单篇 URL 编译仍正常 |

## 附录：核心链路图

### V1.0-alpha 链路（保留）

```
Source（配置）
  → FetchRun（探测执行）
    → SourceItem（扁平的 URL 列表）
      → InsightCard（用户判断）
        → 完整报告 / 行动任务
```

### V1.0-beta 完整目标链路

```
SourceEntry（配置层）
  → FetchRun（探测执行，带状态跟踪）
    → CandidateItem（候选池，元数据级）
      ↓ 用户选择 / 自动过滤
      SourceItem（编译对象，完整 URL）
        ↓ 编译
        InsightCard（单篇洞察）
          ↓ 聚类（V2）
          SignalCluster（主题级）
            → ClusterInsightCard
              → TrendRecord
                → ActionTask
```

### V1.0-beta 第一阶段落地链路

```
SourceEntry
  → FetchRun
    → CandidateItem / SourceItem 过渡
      → InsightCard（单篇编译继续工作）
```
