# V1.0-beta 架构决策

**Date:** 2026-06-08
**Stage:** V1.0-alpha → V1.0-beta 架构分层设计

---

## 1. 架构总原则

### 1.1 分层不变原则

核心业务逻辑必须沉入 `application / domain / adapters / repositories` 分层。

**Web 只是第一个客户端，不是系统本体。**

所有业务能力最终都应可以通过 JSON API 调用，不强依赖 Web UI。

### 1.2 依赖方向

```
presentation (templates/routes)
  → application (services/use_cases)
    → domain (entities/value_objects)
      → adapters (http clients, LLM clients)
        → infrastructure (DB, cache, file system)
```

**路由层只处理请求和响应，不写业务逻辑。**

### 1.3 Adapter 约束

- Adapter **不直接写 DB**，不调用 LLM 直接返回结果
- Adapter 只负责与外部系统交互（HTTP 请求、LLM 调用、文件系统）
- 结果由 Application Service 编排后写入 Repository

---

## 2. Presentation Layer

**职责：** 接收 HTTP 请求，调用 Application Service，返回渲染后的 HTML 或 JSON。

### 2.1 当前实现

- FastAPI + Jinja2Templates
- 所有路由在 `app/main.py`
- HTML 模板在 `app/templates/`

### 2.2 演进规则

- **新增页面** → 新增 template，不修改现有 template
- **局部刷新** → 使用 HTMX 渐进增强，不全页刷新
- **完整页面跳转** → 仅用于导航级跳转（列表 → 详情）
- **高频操作**（候选池浏览、状态更新）→ 必须 HTMX 局部刷新
- **JSON API** → 可选，用于 App / Desktop / CLI 调用

### 2.3 禁止事项

- 不允许在 template 中写 SQL
- 不允许在 template 中调用 LLM
- 不允许在 template 中直接读写文件系统
- 核心业务不允许依赖 HTMX（HTMX 只是增强层）

---

## 3. Application Layer

**职责：** 编排业务用例，负责 transaction boundary，不直接处理 HTTP。

### 3.1 目录结构

```
app/application/
  candidate_pool/
    services.py      # 候选池业务编排
    use_cases.py     # 具体用例（batch_compile, batch_ignore）
  sources/
    services.py      # 来源管理服务
    use_cases.py     # 添加来源、同步配置
  insight_cards/
    services.py      # 编译服务（复用现有 compile_url）
```

### 3.2 命名规范

```python
# 服务类
class CandidatePoolService:
    def list_candidates(...) -> Page[CandidateItem]: ...

# 用例函数
def batch_compile_candidates(candidate_ids: list[int]) -> list[int]:
    """返回生成的 InsightCard IDs"""
    ...
```

### 3.3 Transaction Boundary

每个 HTTP 请求对应一个 transaction。

```python
@app.post("/candidate-pool/batch-compile")
def batch_compile(pool_ids: list[int] = Body(...)):
    result = batch_compile_candidates(pool_ids)  # transaction inside
    return {"ok": True, "card_ids": result}
```

## 5. Adapter Layer

**职责：** 与外部系统交互（HTTP 抓取、LLM 调用、文件系统）。所有网络调用走这里。

### 5.1 当前 Adapter

| Adapter | 文件 | 用途 |
|---------|------|------|
| `minimax_anthropic.py` | `app/llm/providers/` | MiniMax / Anthropic LLM 调用 |
| `openai_compatible.py` | `app/llm/providers/` | OpenAI-compatible API 调用 |

### 5.2 V1.0-beta 新增 Adapter

```python
# app/adapters/http/fetch_runner.py
class FetchRunner:
    """执行来源探测，调用 RSS 或 HTML 探测器"""

    def run(self, source_entry: SourceEntry) -> FetchRunResult:
        strategy = source_entry.fetch_strategy
        if strategy == FetchStrategy.RSS:
            return self._run_rss(source_entry)
        elif strategy == FetchStrategy.HTML_INDEX:
            return self._run_html_index(source_entry)
        else:
            raise ValueError(f"Unknown strategy: {strategy}")


# app/adapters/content/parser.py
class ContentParser:
    """解析探测结果，提取元数据"""

    def parse_feed(self, xml_content: str) -> list[CandidateMetadata]: ...
    def parse_html_index(self, html_content: str, selectors: dict) -> list[CandidateMetadata]: ...
```

### 5.3 Adapter 约束

- Adapter **不写 DB**，只返回数据
- Adapter **不调用 Repository**
- Adapter 所有方法必须可 mock（用于测试）
- 所有外部调用必须有 timeout（默认 30s，超时不卡死 UI）

## 6. Infrastructure Layer

**职责：** 数据库访问（Repository）、缓存、文件系统。

### 6.1 Repository 模式

```python
# app/infrastructure/repositories/candidate_item_repository.py
class CandidateItemRepository:
    def __init__(self, db: Session): ...

    def list_(self, filters: CandidateFilters, pagination: Pagination) -> Page[CandidateItem]: ...

    def create_batch(self, items: list[CandidateItem]) -> list[int]: ...

    def update_status(self, ids: list[int], status: CandidateStatus) -> None: ...

    def get_by_content_hash(self, content_hash: str) -> CandidateItem | None: ...
```

### 6.2 Repository 约束

- Repository 只做数据库访问，不做业务判断
- Repository 不调用 Adapter
- Repository 不调用 LLM
- 所有查询必须支持分页

## 7. 多端扩展原则

### 7.1 Web First but API Ready

当前 Web (FastAPI + Jinja2) 是第一个客户端，但核心业务必须可通过 JSON API 调用。

```python
# 未来支持 App / Desktop / CLI
# 只需新增 presentation 层，不改 application / domain / adapters / infrastructure

@app.post("/api/candidate-pool/batch-compile")
def api_batch_compile(request: BatchCompileRequest):
    result = batch_compile_candidates(request.candidate_ids)
    return {"ok": True, "card_ids": result}
```

### 7.2 多端架构

```
Web Client (Jinja2 + HTMX)
App Client (React Native / Swift)
Desktop Client (Electron / Tauri)
CLI Client (click / typer)
        ↓ JSON API
    FastAPI Router
        ↓
    Application Service
        ↓
    Domain Entities + Rules
        ↓
    Adapter (HTTP / LLM / File)
        ↓
    Infrastructure (DB / Cache)
```

## 8. UI 技术选型

### 8.1 当前栈

- FastAPI + Jinja2Templates
- Vanilla CSS (style.css)
- No JavaScript framework

### 8.2 V1.0-beta Web 演进

| 场景 | 技术 | 原因 |
|------|------|------|
| 局部刷新 | HTMX | 轻量，无需构建工具，渐进增强 |
| 实时状态 | Server-Sent Events (SSE) | FetchRun 状态进度推送 |
| 分页 | HTMX + URL 参数 | 保持可分享链接 |
| 表单 | FastAPI Form + HTMX | 保持简单性 |
| 交互反馈 | CSS + minimal JS | 不引入 React/Vue |

### 8.3 禁止在 V1.0-beta 引入

- ❌ React / Vue / Angular
- ❌ Next.js
- ❌ TypeScript 前端编译
- ❌ Webpack / Vite
- ❌ Tailwind CSS

**原因：** 增加复杂度，需要重新学习，需要构建工具链。当前团队（和个人维护）不需要这些。

## 9. 性能与内存约束

### 9.1 探测阶段性能

| 规则 | 说明 |
|------|------|
| 只抓元数据 | 探测阶段不抓全文，只取 title / URL / published_at |
| PDF 延迟解析 | PDF 全文在用户确认编译后才抓取 |
| 候选池不加载正文 | CandidateItem 只存元数据 |
| timeout 控制 | 所有 HTTP 调用 30s 超时 |
| max_items 限制 | 单次探测最多抓 N 条（可配置，默认 100） |
| max_pages 限制 | HTML Index 最多翻 N 页（可配置，默认 5） |
| retry 限制 | 失败最多重试 2 次 |

### 9.2 批量任务性能

| 规则 | 说明 |
|------|------|
| 长任务落 FetchRun | 探测和编译作为异步任务，不卡 UI |
| 分页加载 | 候选池必须分页，不允许全量加载 |
| 增量更新 | 基于 content_hash 去重，跳过重复条目 |

### 9.3 UI 状态规范

所有批量操作必须支持以下状态：

```
loading → success
        → partial_success（部分成功）
        → failed（全部失败）
        → manual_required（需要用户确认，如 PDF）
```

## 11. 禁止事项

以下在 V1.0-beta 开发中明确禁止：

| 禁止项 | 原因 |
|--------|------|
| 不引入 React / Vue / Next.js | 增加复杂度，不需要 |
| 不引入向量数据库 | 当前阶段用不上 |
| 不重构现有 alpha 流程 | 保持 alpha 稳定可用 |
| 不做用户认证系统 | 单用户本地使用 |
| 不做 SaaS 部署 | 个人工具定位 |
| Adapter 不直接写 DB | 违反分层原则 |
| Repository 不调用 LLM | 违反分层原则 |
| 路由层不写业务逻辑 | 违反分层原则 |
| 核心业务不依赖 HTMX | HTMX 只是增强层 |
| 不提交 `.env` | 密钥安全 |
| 不提交 `data/*.db` | 数据文件 |
| 不提交 `__pycache__/` | 缓存文件 |
