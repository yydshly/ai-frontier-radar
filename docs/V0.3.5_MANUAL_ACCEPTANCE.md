# V0.3.5 中文用户端到端人工验收

## 目标

验证系统是否能帮助英语能力有限的中文用户，从英文 AI 前沿资料中发现 → 选择 → 编译 → 阅读中文 InsightCard。

本轮**不**做批量翻译，不做自动推荐，不做后台队列。
本轮只确认：**当一个用户第一次打开 `/source-items` 时，他/她知道下一步该点哪里。**

---

## 前置条件

- 已安装 Python 3.10+ 与项目依赖（`pip install -r requirements.txt`）
- 已复制 `.env.example` 为 `.env`（如已有可跳过）
- 至少有 LLM Profile 配置（即使没有 API Key，也可以验证到「failed card」状态）
- 数据库可写

如果你只是想跑通页面、UI 流程，**没有 API Key 也完全可以**——系统会生成 `failed` InsightCard，详细说明失败原因。

---

## 第一步：清理或备份旧数据库

V0.3.3 之前探测过 Hugging Face 的话，你的旧数据库里可能仍然有这类**历史脏数据**：

```text
https://huggingface.co/blog?p=56
https://huggingface.co/blog?p=2
https://huggingface.co/blog?p=1
https://huggingface.co/blog?p=0
https://huggingface.co/blog
https://huggingface.co/posts
```

这些不是文章详情页，是分页/列表页。

如果你想立即看干净数据，**先备份再重建数据库**（Windows PowerShell）：

```powershell
# 备份
Copy-Item data\ai_frontier_radar.db data\ai_frontier_radar.backup.db

# 重建
Remove-Item data\ai_frontier_radar.db
```

不想重建也行——V0.3.4 提供了一个**只读检查脚本**可以识别这些历史脏数据：

```bash
python scripts/check_listing_source_items.py --source-key huggingface_blog
```

输出形如：

```text
Potential listing SourceItems:
  #44  huggingface_blog  discovered https://huggingface.co/blog?p=56
  #43  huggingface_blog  compiled   https://huggingface.co/blog?p=2
  #1   huggingface_blog  discovered https://huggingface.co/blog
Total suspected listing items: 6
```

注意：脚本**只检查，不删除**。是否清理由你决定。

## 第二步：探测 huggingface_blog

启动探测（Windows PowerShell）：

```powershell
python scripts\acceptance_probe_sources.py --repeat 2 --timeout 15 --html-source huggingface_blog
```

你应该看到类似输出：

```text
[OK] HTML huggingface_blog: status=success, found=38, new=38, updated=0, failed=0

  Top discovered SourceItems (huggingface_blog - HTML):
    - Shipping a Trillion Parameters With a Hub Bucket
      https://huggingface.co/blog/delta-weight-sync
    - Harness, Scaffold, and the AI Agent Terms
      https://huggingface.co/blog/agent-glossary
    ...
```

**关键观察**：

- 全部 URL 形如 `/blog/{slug}`（文章详情页）
- 没有 `/blog?p=2`、`/blog?page=...`、`/blog?sort=...`
- 第二次运行 `items_new=0`、`items_updated=38`（幂等）

如果你看到 `items_found=0`，说明 huggingface_blog 探测失败——可能网络问题或被反爬，请检查 `error_message`。

## 第三步：打开 SourceItem 列表

启动 web 服务（Windows PowerShell）：

```powershell
uvicorn app.main:app --reload --port 8779
```

浏览器打开：

```text
http://127.0.0.1:8779/source-items?source_key=huggingface_blog
```

你应该看到：

1. 顶部有 **「如何使用这个页面？」** 中文引导（蓝色框）
2. 紧接其下是 **「V0.3.3 已过滤新的分页/列表 URL」** 黄色提示
3. 表格列：ID、来源、标题、URL、状态、发布时间、首次发现、最近发现、关联卡片、**推荐操作**
4. 每行有 4 种状态对应动作：
   - `discovered` → **「进入详情并编译」**（蓝色按钮）
   - `compiled` → **「查看中文卡片」**（绿色按钮）
   - `failed` → **「查看失败原因 / 重试」**（红色边框按钮）
5. URL 单元格完整显示，可横向滚动

**关键判断**：

- 看到 `/blog/{slug}` 类型 URL 的条目 → 是文章详情页 → 可以继续下一步
- 看到 `/blog?p=...` 或 `/blog`（没有 slug）→ 是历史脏数据，**跳过**

---

## 第四条：选择一条文章

挑选标准的优先级：

```text
1. URL 形如 /blog/some-article-title
2. 标题像具体文章，而不是分类页/分页页
3. 不选 /blog?p=2、/blog?page=...、/blog、/posts
```

如果你英语能力有限，看不懂英文标题，可以：

- 点开原文（URL 列的蓝色链接）快速扫一眼
- 选看起来像「具体技术/产品发布」的，跳过「十大盘点」「年度回顾」类聚合

点开条目 ID 链接（或推荐操作列的「进入详情并编译」），进入详情页。

---

## 第五步：进入详情页并编译

详情页（`/source-items/{id}`）应该有：

1. 顶部蓝色框「这是什么？」说明这是英文前沿候选资料
2. 完整信息表（标题、URL、状态等）
3. 状态对应的操作区：
   - `discovered`：底部有「编译为 InsightCard」按钮 + 灰色帮助文字「适合在你看不懂英文标题、但想知道它是否值得关注时使用。」
   - `failed`：显示错误信息 + 「重新编译」按钮 + 「失败可能来自：网页无法访问、正文抽取失败、API Key 缺失、LLM 调用失败」说明
   - `compiled`：提示已编译，链接到 InsightCard

**点击「编译为 InsightCard」按钮**。

页面会跳转（303），回到详情页，并显示：

- 状态从 `discovered` 变为 `compiled` 或 `failed`
- 关联卡片行从 `-` 变成 `#X`
- 底部出现 InsightCard 链接

---

## 第六步：查看中文 InsightCard

点击关联卡片行（`#X`）或详情页底部的「查看卡片」链接，进入 `/cards/{id}`。

你应该看到 InsightCard 包含：

```text
- 中文摘要
- 关键事实
- 技术洞察
- 产品机会
- 风险
- 行动建议
- 相关性评分
```

确认中文可读、不是英文搬运、结构合理。

## 常见问题

### Q1：为什么我看到 `/blog?p=2` 这种记录？

这是 V0.3.3 之前的旧数据。V0.3.3 已经能阻止新的分页 URL 入库，但不会自动清理你的旧数据库。

```powershell
# 方案 A：检查后手动备份重建
python scripts\check_listing_source_items.py --source-key huggingface_blog
Copy-Item data\ai_frontier_radar.db data\ai_frontier_radar.backup.db
Remove-Item data\ai_frontier_radar.db
python scripts\acceptance_probe_sources.py --repeat 2 --timeout 15 --html-source huggingface_blog

# 方案 B：忽略它们，只在列表页通过筛选跳过
# /source-items?source_key=huggingface_blog&q=具体slug
```

### Q2：为什么编译失败？

打开详情页的「错误信息」行。常见原因：

```text
- 原文 URL 无法访问（403/404/超时）
- 原文正文太短或被反爬挡住
- .env 中没有配置 MINIMAX_API_KEY
- LLM 调用失败（rate limit、网络问题）
```

修复后回到详情页点「重新编译」即可。

### Q3：为什么生成的是 failed card？

V0.3.2 的设计：编译流程任何一步失败，都会创建一个 `failed` 状态的 InsightCard 并把错误信息写进去。

这不是 bug，是可排查的状态。failed card 在 `/cards` 列表里可以看到，点进去能看到完整 `error_message`。

### Q4：没有 API Key 怎么办？

如果你只想验收 UI 流程（页面跳转、状态变化、推荐操作列），**没有 API Key 也完全可以**。

如果你想看到真实的「中文摘要 + 关键事实 + 行动建议」，需要：

```bash
# 1. 注册 MiniMax 账号（或其他支持的 LLM 提供方）
# 2. 拿到 API Key
# 3. 写入 .env
MINIMAX_API_KEY=sk-...

# 4. 重新探测并编译
python scripts\acceptance_probe_sources.py --repeat 2 --timeout 15 --html-source huggingface_blog
# 然后到 /source-items 选条目，点击「编译为 InsightCard」
```

### Q5：怎么重建本地数据库？

```powershell
# 1. 停止 uvicorn
# 2. 备份
Copy-Item data\ai_frontier_radar.db data\ai_frontier_radar.backup.db

# 3. 删除
Remove-Item data\ai_frontier_radar.db

# 4. 重新探测
python scripts\acceptance_probe_sources.py --repeat 2 --timeout 15 --html-source huggingface_blog
```

下次启动 uvicorn 时，数据库会被自动重建（`init_db()`）。

### Q6：编译后的 InsightCard 不理想怎么办？

当前阶段 InsightCard 由 LLM 生成，质量取决于：

```text
- LLM 本身的理解能力
- Prompt 模板
- 原文内容质量
- API Key 配额
```

本轮**不做**自动重试、自动评分、人工反馈入口。
如果你觉得 InsightCard 不理想：

- 检查原文是否有正文（不是登录墙/纯图片）
- 检查 LLM Profile 是否用对了
- 重新编译（重新调用 LLM）

## 验收清单

完成上述步骤后，请确认：

```text
[ ] /source-items 页面有中文「如何使用这个页面？」引导
[ ] 表格有「推荐操作」列
[ ] discovered 条目显示「进入详情并编译」并能跳转
[ ] compiled 条目显示「查看中文卡片」并能跳转到 /cards/{id}
[ ] failed 条目显示「查看失败原因 / 重试」并能跳转
[ ] 详情页有「这是什么？」中文说明
[ ] 编译按钮附近有「适合在你看不懂英文标题…」等中文提示
[ ] 编译后能看到中文摘要、关键事实、技术洞察、产品机会
[ ] 不需要懂英文也能理解页面在做什么
```

任意一项打不到，说明 V0.3.5 中文优先体验目标没有达成，请回到 README V0.3.5 章节对照检查。

## 下一步

本轮验收通过后，可以进入下一阶段的功能探索（不在本轮范围）：

- 批量编译（按 source_key 一次性编译多条）
- 轻量队列（后台自动处理失败的 SourceItem）
- 标题翻译（避免 LLM 额度浪费）
- 相关性推荐（基于用户标签的简单排序）

但 V0.3.5 暂不做这些——先确保**单条端到端体验闭环**。
