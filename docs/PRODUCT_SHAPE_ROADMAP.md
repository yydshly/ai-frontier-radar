# AI Frontier Radar 产品形态路线

## 1. 当前产品形态

当前是**个人 AI 前沿资料工作台**，不是公开 SaaS。

核心特征：
- 单用户本地运行
- SQLite 本地数据库
- 手动触发编译（非定时任务）
- 需要用户自己配置 API Key
- 可演示的工作台雏形，不是面向最终用户的完整产品

## 2. 当前用户场景

目标用户画像：英语阅读能力有限，但关注全球 AI 前沿进展的个人用户。

用户希望：
- 看懂英文资料（不需要逐字读完）
- 保留英文原意（不是机翻）
- 获得中文解释（通俗解说）
- 判断和自己方向的关系（相关性判断）
- 转成可执行的任务（行动导出）

## 3. 当前核心体验

用户打开系统后的完整路径：

```
首页工作台
    ↓
待编译资料收件箱（/source-items）
    ↓
点击感兴趣的文章
    ↓
编译为中文 InsightCard
    ↓
查看中英双语理解（可选）
    ↓
做用户判断（值得关注 / 与我有关 / 稍后再看 / 暂时忽略 / 转成行动）
    ↓
导出为 Markdown 行动任务
    ↓
交给 AI 执行模型或保存到本地
```

## 4. 后期产品形态需要调整的方向

当前系统的形态是"页面集合"：来源列表、发现条目列表、卡片列表、卡片详情、导出页。

后期应该演化为"工作流产品"：

### 发现层
- 今天发现了什么新的 AI 前沿文章
- 哪些来源更新了
- 哪些值得进一步编译

### 理解层
- 英文核心：原文的主旨、关键主张、证据点
- 中文解说：帮助理解的通俗中文说明
- 技术洞察：这篇文章的技术深度分析
- 产品机会：可以做什么

### 行动层
- 标记处理状态
- 转成具体任务
- 导出为可执行文档
- 后续沉淀为个人知识库

## 5. 不建议现在做的形态

以下方向在当前阶段不建议做，原因不是"不能做"，而是"现在做会稀释核心价值"：

| 不建议做的方向 | 原因 |
|---------------|------|
| 多用户 SaaS | 单用户 MVP 还没走完，多用户引入权限、隔离、注册等一系列复杂度 |
| 浏览器插件 | 当前没有离线能力，插件价值有限 |
| 移动 App | 主要场景是电脑前阅读，移动端不是核心场景 |
| 公众号发布 | 内容生产者的诉求，不是这个工具的定位 |
| 全网爬虫 | 质量不可控，会稀释来源的专业性 |
| 复杂推荐系统 | 没有用户行为数据，推荐是无根之木 |
| 向量数据库 | 没有足够的 embedding 需求，SQLite 足够 |
| 知识图谱 | 需要大量人工标注，当前阶段 ROI 低 |

## 6. 可选演化方向

以下方向是可能的演化路径，需要根据用户反馈选择：

### 方向 A：个人本地工作台（当前路径）

继续完善个人工作台体验：
- 更丰富的统计和可视化
- 更好的卡片组织和筛选
- 笔记本/标签功能
- 更完整的周报生成

**适合**：已经有固定使用习惯的个人用户

### 方向 B：Notion / Markdown 知识沉淀器

导出到外部知识管理工具：
- Notion API 同步
- Obsidian 双向链接
- 更丰富的 Markdown 模板
- 本地文件自动归档

**适合**：已有知识管理习惯的用户

### 方向 C：AI 前沿周报生成器

在个人工作台基础上增加：
- 每周汇总已编译的卡片
- 生成结构化周报（本周重要发现、与我有关的内容、行动项）
- 自动发送到邮箱（可选）

**适合**：希望定期回顾的用户

### 方向 D：独立开发者机会雷达

专注独立开发者和 startup 场景：
- 追踪 AI 技术动态
- 分析竞品和新技术
- 生成技术调研报告
- 转化为可执行的产品 idea

**适合**：独立开发者、技术创业者

### 方向 E：技术报告编译器

面向更专业的技术人群：
- 支持更长文章的编译
- 更丰富的结构化字段
- 支持技术论文格式
- 引用和来源追溯

**适合**：研究员、学者、技术决策者

## 7. 下一阶段建议

### V0.9：完整 Markdown 报告导出

当前 `build_action_markdown()` 主要导出的是行动任务，不是完整报告。

V0.9 的目标是把一张 InsightCard 导出成**完整报告**，包括：

- 英文核心摘要（English Core Summary）
- 原文主张（Original Key Claims）
- 证据点（Key Evidence Points）
- 术语表（Key Terms EN-ZH）
- 中文解说（Chinese Explanation）
- 关键事实（Key Points）
- 技术洞察（Technical Insights）
- 产品机会（Product Opportunities）
- 风险（Risks）
- 行动建议（Action Items）
- 保真提示（Fidelity Notes）
- 解读边界（Interpretation Boundary）
- 用户判断（CardDecision）
- AI 执行引导问题

这样产品会更像"前沿资料编译器"，而不只是"任务草稿生成器"。

### V1.0-alpha：可演示的最小完整闭环

在 V0.9 基础上：
- 固定一条演示主流程（演示脚本，从探测到导出一步完成）
- 清理页面入口（首页只保留核心路径）
- 提供完整使用说明（README + 文档）
- 优化用户体验（减少不必要的点击）

V1.0-alpha 的目标：让新用户能在 10 分钟内理解产品价值并完成一次完整体验。

## 8. 当前阶段的核心建议

**不要急着做公开产品，先做个人工作台闭环。**

MVP 阶段的核心验证：
1. 来源配置能否稳定发现高质量文章
2. 编译链路能否生成有价值的中文洞察
3. 用户判断和导出能否形成真正的行动闭环

这三个问题回答清楚之后，再考虑产品形态演化。

现在做太多形态探索会分散注意力。当前最重要的两件事：
- 让演示流程稳定可跑
- 让真实 LLM 输出质量达标
