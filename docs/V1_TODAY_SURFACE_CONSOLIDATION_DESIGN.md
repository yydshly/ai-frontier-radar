# 今日信息消费面收敛设计（B1）

> 状态：设计稿（待评审，未实施）
> 关联：今日雷达质量治理（[V1_BETA_16_TODAY_RADAR_QUALITY](V1_BETA_16_TODAY_RADAR_QUALITY.md)）、
> 条目状态模型（[V1_ITEM_STATE_MODEL_DESIGN](V1_ITEM_STATE_MODEL_DESIGN.md)）、
> 相关性统一（[V1_RELEVANCE_UNIFICATION_DESIGN](V1_RELEVANCE_UNIFICATION_DESIGN.md)）

## 1. 背景

围绕「今天新增了什么」目前存在 5 个独立的呈现面：

| # | 面 | 路由 | 形态 | 是否调用 LLM |
|---|----|------|------|-------------|
| 1 | 今日新增简报 | `GET /radar/today/briefing` | 按来源分组的扁平列表（确定性） | 否 |
| 2 | 今日雷达（主页） | `GET /radar/today` | 工作台：分区目录 + 卡片 + 阅读面板 | 否（生成动作另触发） |
| 3 | 今日核心报告 | `GET /radar/daily-report` | 规则排序的要点卡片；可显式生成 LLM 核心报告 | 可选 |
| 4 | 语音简报 | `GET /radar/daily-report/broadcast` | 播报脚本 + 播放器（MiMo TTS） | 否（TTS 另触发） |
| 5 | 侧栏摘要模块 | `today_summary_panel`（嵌于 #2） | 主页侧栏中的「核心报告 + 最新音频」小模块 | 否 |

## 2. 问题定义

这 5 个面**并非 5 份重复**，而是同一批数据在不同**消费阶段**的形态（见 §3）。真正的问题是：

- **P-B1-1 缺信息层级**：用户落到任一页面，无法感知 5 个面的关系与推荐使用路径。
- **P-B1-2 命名同质化**：「雷达 / 新增简报 / 核心报告 / 可读简报 / 语音简报」彼此难以区分、难以记忆。
- **P-B1-3 底层「今日」各算各的**：每个面各自重建一遍今日数据（属代码重复，可与产品形态解耦治理）。

> 反目标（Non-goals）：**不**把 5 个面合并为单一页面。不同消费模式（扫读 / 加工 / 读结论 / 收听）合并只会产生臃肿且谁都不顺手的页面。

## 3. 设计原则与心智模型

**原则**：保留各消费模式的独立性，统一其**层级、命名与导航**；底层数据构建去重作为独立工程项。

**心智模型**（建议对用户呈现的主线）：

```
速览（简报）  →  工作台（雷达）  →  精炼（核心报告）  →  收听（语音）
 今天来了啥      逐条读 / 加工        今天最该看的几条      把结论读给你听
```

侧栏摘要模块（#5）定位为「工作台内指向精炼/收听的入口」，不是独立一站。

## 4. 命名体系（建议）

| 现名 | 建议名 | 路由（不变） | 一句话定位 |
|------|--------|-------------|-----------|
| 今日新增简报 | **今日速览** | `/radar/today/briefing` | 今天新增内容的完整清单，确定性、可扫读 |
| 今日雷达 | **今日工作台**（主名仍可叫「今日雷达」） | `/radar/today` | 逐条阅读、补摘要、生成洞察的主场 |
| 今日核心报告 | **今日要点** | `/radar/daily-report` | 系统精炼出的当日要点 |
| 语音简报 | **今日收听** | `/radar/daily-report/broadcast` | 把当日要点转成可听的播报 |
| 侧栏摘要模块 | （沿用）今日要点·入口 | 嵌于主页 | 工作台内指向要点/收听的预览入口 |

> 命名为建议项，**最终以评审确认为准**（见 §8 待决策）。路由保持不变，避免破坏既有链接与书签。

## 5. 导航设计

为四个「今日」站点提供一致的**子导航**（顶部 tab 或面包屑），在每个面顶部固定呈现：

```
[今日速览] · [今日工作台] · [今日要点] · [今日收听]
```

- 当前所在站点高亮。
- 顺序即推荐心智路径（左→右：从原料到结论）。
- 由一个共享模板片段（如 `partials/today_subnav.html`）渲染，避免四处复制。

## 6. 分阶段落地

### 阶段一：统一导航 + 命名（推荐先做，低风险）

- **范围**：新增 `partials/today_subnav.html`；在 4 个今日模板顶部引入；按 §4 调整展示文案/标题。
- **不变**：路由、后端逻辑、数据、URL 契约一律不变；纯模板 + 文案。
- **风险**：低（仅 UI 文案与导航）。需 UI 改动签字。
- **验收**：4 个页面均渲染 200；每页含子导航且当前项高亮；现有验收用例（标签/文案断言）随命名同步更新；新增 quick_test 守护：四页均引用 `today_subnav` 且导航项齐全。

### 阶段二（可选）：将「今日速览」并入工作台视图切换

- **动机**：简报本质是工作台同一批文章的扁平列表。
- **范围**：在工作台增加「列表 / 分区」视图切换；简报作为工作台的一个视图模式；`/radar/today/briefing` 保留为兼容跳转或薄包装。页面数 5 → 4。
- **风险**：中（涉及主页交互）。需明确产品偏好——部分用户偏好简报独立成页。
- **验收**：工作台两种视图均可用；简报内容与旧独立页一致（diff 证明条目集合相同）。

### 阶段三（与产品解耦）：共享「今日上下文」构建器

- **动机**：消除 5 个面各自重建今日数据的重复（P-B1-3），延续 P 系列性能治理。
- **范围**：抽出 `build_today_context()`（窗口选 item + 候选 + 报告输入指纹等），供各面复用；各面只取所需切片。
- **风险**：中（行为敏感）。按既有纪律：对会改行为的步骤先出 before/after diff 证明产出一致，再实施。
- **验收**：各面渲染产出逐项不变（快照对比）；quick_test/acceptance 全绿。

## 7. 测试与验收策略

沿用本分支既有纪律：
- 行为可证明不变的步骤 → 快照/全量 diff 证明 + 测试锁定。
- 会动 UI 的步骤 → 显式签字；新增 quick_test 守护锁定导航/命名结构。
- 既有 2 个 broadcast 预存失败与本设计无关，不计入。

## 8. 待决策项（需产品确认）

1. **命名**：§4 建议名是否采纳？尤其主页主名保留「今日雷达」还是改「今日工作台」。
2. **简报去留**（阶段二）：是否将「今日速览」并入工作台，还是保留独立页？
3. **子导航形态**：顶部 tab 还是面包屑；是否在侧栏也同步呈现。
4. **阶段三时机**：是否与阶段一/二并行，还是延后作为纯代码治理。

## 9. 建议执行顺序

1. 先落 **阶段一**（导航 + 命名）——投入小、风险低、直接解决「分不清」核心痛点。
2. 视 §8.2 决策决定是否做 **阶段二**。
3. **阶段三** 随时可独立进行，纯代码收益。
