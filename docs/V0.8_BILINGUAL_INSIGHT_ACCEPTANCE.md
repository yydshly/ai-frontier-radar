# V0.8 中英双语 InsightCard 验收

## 目标

让用户在不强读英文全文的情况下，既能看到英文核心内容，又能获得中文解释。

## 为什么要做

只输出中文会有曲解风险；只输出英文对用户阅读成本高；中英双语结构可以兼顾保真和理解。

---

## 验收命令

### Mock 模式（默认，不调用真实 LLM，验证链路）

```bash
python scripts/acceptance_bilingual_report.py --isolated-db --mock
```

### 真实模式（需要 API Key）

```bash
python scripts/acceptance_bilingual_report.py --isolated-db
```

### 保留 DB 方便排查

```bash
python scripts/acceptance_bilingual_report.py --isolated-db --mock --keep-db
```

## 验收步骤

### 1. 创建测试卡片

脚本自动创建一个 completed InsightCard。

### 2. 生成双语报告

- 方式 A（mock 模式）：脚本自动调用 `build_mock_bilingual_report()`
- 方式 B（真实 LLM）：POST `/cards/{card_id}/bilingual-report`

### 3. 验证页面包含

打开 `/cards/{card_id}`，确认显示：

- 🌐 中英双语核心理解（区域标题）
- English Core Summary
- Original Key Claims
- Key Evidence Points（如果有）
- Key Terms EN-ZH 表格（如果有）
- 中文解说
- ⚠️ 保真提示
- 📌 解读边界

### 4. 验证无双语报告时显示生成入口

对于没有双语报告的卡片，确认显示：
- "还没有生成中英双语核心理解"
- "🌐 生成中英双语报告" 按钮

### 5. 导出 Markdown

访问 `/cards/{card_id}/export-markdown`，确认包含：
- English Core Summary
- Original Key Claims
- 中文解说
- 保真提示与解读边界

## inspect_bilingual_report_quality() 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `english_summary_present` | bool | english_core_summary 非空 |
| `english_key_claims_count` | int | english_key_claims 数组长度 |
| `english_evidence_points_count` | int | english_evidence_points 数组长度 |
| `key_terms_count` | int | key_terms 数组长度 |
| `chinese_explanation_present` | bool | chinese_explanation 非空 |
| `fidelity_notes_present` | bool | fidelity_notes_zh 非空 |
| `interpretation_boundary_present` | bool | interpretation_boundary_zh 非空 |
| `passed_minimum_quality` | bool | 所有必需字段非空且 claims>=2 |
| `warnings` | list[str] | 质量警告列表 |

### 通过标准

```
english_core_summary 非空
english_key_claims >= 2
chinese_explanation 非空
fidelity_notes_zh 非空
interpretation_boundary_zh 非空
```

## 页面保真原则

1. **英文内容不是装饰**：让用户看到原文主旨，不做中文化发挥
2. **中文解说不能替代原文事实**：页面明确区分原文事实 / 模型解读 / 个人建议
3. **不做长篇原文复制**：英文部分是模型对原文的忠实英文概括
4. **不确定内容要提示**：产品机会和行动建议要标记为"模型推论"
