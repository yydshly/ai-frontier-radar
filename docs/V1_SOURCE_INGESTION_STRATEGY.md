# 信息来源获取策略阶梯（P-001 设计）

> 设计先行文档。本文件定义来源获取方式的优先级阶梯、能力矩阵、回退规则和
> 受控边界，供 P-002（来源工作台）与 P-004（自定义来源接入）复用。
> 本阶段不改抓取代码、不接爬虫、不调用 LLM。

## 1. 核心原则

> **能用轻量方式拿到结构化数据，就不要用重量方式。**

获取一个来源的"有效信息"时，按成本与脆弱度从低到高选择策略，
优先选最轻、最稳定、最结构化的方式；只有上层不可行时才下沉到更重的方式。

```
RSS / Atom / JSON Feed   ← 首选（结构化、稳定、低成本）
        ↓ 不可用
官方 API / sitemap.xml    ← 次选（结构化、需适配）
        ↓ 不可用
HTML index 解析           ← 补充（已实现，半结构化）
        ↓ 不可用
单篇 URL 抓取             ← 点抓（用户指定具体文章）
        ↓ 不可用
渲染型爬虫 / 变更检测      ← 后置（重、脆弱、需显式开启）
        ↓ 不可用
人工 / PDF / newsletter   ← 外部录入（人工兜底）
```

## 2. 策略能力矩阵

| 策略 key | 层级 | 结构化 | 成本 | JS 站 | 当前状态 | 默认自动调度 |
|----------|------|--------|------|-------|----------|--------------|
| `rss` | 轻量 | 高 | 低 | — | ✅ 已支持 | ✅ 是 |
| `json_feed` | 轻量 | 高 | 低 | — | 🔜 预留 | 拟支持 |
| `sitemap` | 轻量 | 中 | 低 | — | 🔜 预留 | 拟支持 |
| `api` | 轻量 | 高 | 低-中 | — | 🔜 预留（按来源适配） | 拟支持 |
| `html_index` | 中量 | 中 | 中 | ❌ | ✅ 已支持 | ✅ 是 |
| `single_url` | 中量 | 低 | 中 | 部分 | 🔜 预留 | 拟支持 |
| `crawler` | 重量 | 低 | 高 | ✅ | ⛔ 后置 | ❌ 否（需显式开启） |
| `change_detect` | 重量 | 低 | 高 | ✅ | ⛔ 后置 | ❌ 否 |
| `pdf` | 外部 | 中 | 中 | — | ⛔ 人工 | ❌ 否（人工录入） |
| `newsletter` | 外部 | 中 | 中 | — | ⛔ 人工 | ❌ 否 |

- **已支持**：当前 `SUPPORTED_STRATEGIES = {"rss", "html_index"}` 已实现并进自动调度。
- **预留**：词表中保留 key，先在设计与校验层定义，待后续阶段实现。
- **后置 / 人工**：重量与外部策略默认**不进**自动调度，需显式开启或人工触发，
  避免成本、封禁、脆弱性扩散到默认链路。

## 3. 回退规则（设计）

每个来源声明一个**首选策略**，可选声明**回退链**。回退是设计意图，
落地时分阶段实现，且必须遵守：

1. 回退只在轻量 / 中量层内自动发生（rss → json_feed → sitemap → html_index）。
2. **不自动回退到重量层**（crawler / change_detect）——必须人工确认或显式开启。
3. 回退不得绕过 running 去重与 due-source 调度。
4. 回退失败保留 FetchRun 失败记录与 error_message，便于来源工作台诊断。

示例（设计意图，非当前实现）：

```yaml
openai_news:
  fetch_strategy: rss
  fallback_strategies: [html_index]   # rss 不可用时降级
```

## 4. "有效信息"的判定

无论用哪种策略，入库的 SourceItem 至少要拿到：

- 标题（title）
- 原文链接（url，去重键）
- 发布 / 发现时间（published_at / first_seen_at）
- 可选：摘要原文 / 正文片段（供后续中文摘要）

拿不到 url 的条目计入 `items_failed`，不污染有效条目（与现有 `rss_probe` 一致）。

## 5. 受控边界

- 重量策略（crawler / change_detect）默认关闭，需独立开关，不进默认自动调度。
- PDF / newsletter 走人工录入路径，不做自动外呼抓取。
- 新策略接入前先扩展 `SUPPORTED_STRATEGIES` 与能力矩阵，再实现 probe；
  due-source 对未实现策略继续判 `unsupported`，不报错、不 500。
- 抓取内容进入 LLM 前的清洗与 prompt injection 防护沿用现有链路。

## 6. 对后续阶段的接口

- **P-002 来源工作台**：用本矩阵把 `fetch_strategy` 翻译成用户可懂的"获取方式"
  文案（如 `rss`→"RSS 订阅"，`html_index`→"网页索引解析"）。
- **P-004 自定义来源接入**：新增来源时，策略选择受本矩阵白名单约束——
  普通用户只能选已支持 / 预留的轻中量策略；重量 / 外部策略需显式开启。

## 7. 本阶段不做

- 不修改 `SUPPORTED_STRATEGIES` 实际集合（仍 `{rss, html_index}`）。
- 不实现 json_feed / sitemap / api / single_url / crawler / pdf 的 probe。
- 不接爬虫、不调用 LLM、不改抓取流程。
