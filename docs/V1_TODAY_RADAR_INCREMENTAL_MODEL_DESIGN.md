# 今日雷达增量与多源展示模型设计

> 状态：设计稿（待评审，未实施）

> [V1_BETA_6_TODAY_RADAR_MAIN_CHAIN_CHECKPOINT](V1_BETA_6_TODAY_RADAR_MAIN_CHAIN_CHECKPOINT.md)（来源层 bootstrap / daily_increment）、
> [V1_BETA_16_TODAY_RADAR_QUALITY](V1_BETA_16_TODAY_RADAR_QUALITY.md)

## 1. 背景：链路通了，但第②层语义错了

围绕「今天新增了什么」，链路分三层：

| 层 | 职责 | 现状 |
|----|------|------|
| ① 来源获取（地基） | RSS 优先 + 兜底探测，可靠拿到信息 | `sources.yaml`→Source→`compute_due_sources`→`rss_probe`/`html_index_probe`→SourceItem。来源层已区分 **bootstrap（首灌）/ daily_increment（按到期增量抓取）** |
| ② 今日雷达集合层 | 决定"给用户看哪些文章" | **滚动 24h 时间窗（`recent_valid_items_query`）+ 50 条截断（`item_limit`）**。无"自上次分析以来"的概念 |
| ③ 上层模块 | 摘要 / 洞察 / 报告 / 速览 / 推荐 | 基于②给出的 item 集合，概念正确 |

**根因在第②层**：它用"滚动窗口 + 50 条截断"，既不是"今天真正新增"，也不是"上次分析后的增量"。来源层好不容易区分了首灌/增量，到显示层被抹平。

## 2. 症状（均由同一根因导出）

实测数据：窗口真实总量约 **2218**，`item_limit=50`。

| 症状 | 机制 |
|------|------|
| S1 徽标 2218 无意义 | 滚动窗口总量，非"增量" |
| S2 单源刷屏、其他源不露面 | `_select_window_items` 按 `last_seen_at/first_seen_at` 倒序后截断，**无按源平衡**；顺序抓取下最后抓的源整片占前列 |
| S3 不该有上限 | `item_limit=50` 硬截断显示与分析范围 |
| S4 首次 / 后续不分 | 显示层无"首次"概念，首灌批和后续混在同一窗口 |
| S5 左侧计数不准（全部 / 各分区） | `section_counts` 从**被截断的 50 条**算：「全部」=`len(items)`≤50，各分区在这 50 条里分类求和≤50，而「今日速览」徽标用未截断 `count()`=2218 → 三套数对不上、且"更新今日新增"后「全部」永远卡 50 |

一句话：**计数与展示都建立在"显示用的前 50 条"上，而非"集合真实全量"上。**

## 3. 目标模型

```
① 来源层：RSS 优先 + 兜底探测 → SourceItem（可靠拿到）
② 今日雷达集合层（重构）：
   增量基准 = 每日时间锚点（默认每天 08:00，确定性计算，无需持久化）
   主集合 = first_seen_at ≥ 最近锚点 的增量；全量显示（无上限）、按文章发现时间倒序 + 按源轮转、计数在全量上算
   每到次日锚点 → 集合自动重置为新一天增量（无需用户手动标记）
   首次 bootstrap 首批的 first_seen_at ≈ 当下，自然落在锚点内 → 与后续统一处理
③ 上层模块：基于②的集合，逻辑不变
```

集合从"滚动窗口（无界）"换成"增量 / 首批（天然有界）"后：S1–S5 一并消解——集合小到可以不截断、可在全量上准确计数、可按源轮转不刷屏，首次/后续由"有无水位线"天然区分。

## 4. 机制设计

### 4.1 每日时间锚点（daily anchor）——增量基准【定版策略】
- 增量以一个**每日固定时间点**为基准（默认每天 08:00；时区可配，面向中国默认 Asia/Shanghai = UTC+8）。
- `anchor(now)` = 不晚于 now 的**最近一次锚点时刻**（若 now 早于今日 08:00，则取昨日 08:00）。**确定性计算，无需持久化、不改 DB schema。**
- 可配置：`RADAR_DAILY_ANCHOR_HOUR`（默认 8）、时区偏移（默认 +8）。DB 时间为 UTC，锚点按本地时刻换算到 UTC 后比较。
- **与抓取解耦**：每天 08:00 的"增量刷新"（来源层 daily_increment 抓取）只负责灌入新文章；展示锚点是确定性的——即使抓取迟到或被手动触发，展示仍"以 08:00 节点为准"。

### 4.2 增量集合
- 主集合 = `recent_valid_items_query` 基础过滤 + `first_seen_at >= anchor(now)`，**不再 `.limit()` 截断**。
- **"新增按文章时间"**：集合内按 `first_seen_at` 倒序（可靠列；`published_at` 为自由文本不可靠，仅作展示不作排序依据）。
- 仅保留一个**安全天花板**（默认 2000，防首灌误配置炸内存），正常一天增量远低于它。

### 4.3 锚点滚动（替代手动水位线）
- 每越过一个锚点（次日 08:00），`anchor(now)` 自动前移 → 集合自然重置为新一天的增量。**无需用户手动「标记已读」。**
- 手动「同步今日新增」只把新文章灌入**当前锚点周期**内，按发现时间出现，不移动锚点。
- 可选增强（后续，非必须）：手动「刷新到此刻」临时把基准设为当前时刻以清空当前增量。

### 4.4 首次 / 空窗
- 首次 bootstrap 首灌文章的 `first_seen_at ≈ 当下`，自然落在"≥ 最近锚点"内 → 全批展示（量由来源层 `max_sources`/`max_items_per_source` 天然限定）。**因此无需单独的"首次"分支**，首次与后续由"锚点 + 文章发现时间"统一处理。
- 首屏仍**默认按来源分组**展示，保证每源露面。
- 锚点后无新增 → 空态提示（如"自今日 08:00 以来暂无新增，可点击同步今日新增"）。

### 4.5 多源公平排序
- 增量集合内：按来源**轮转 (round-robin)** 或 **每源上限 N + 展开全部**，复用「推荐」已有的 `per_source_limit` 思路。
- 保证第一屏每个来源都至少露面，消除 S2。

### 4.6 计数准确性
- `section_counts` 改为在**增量全量**上计算（集合已有界，分类成本可接受）。
- 「全部」「各分区」「今日速览」共用同一集合口径 → 三套数自洽。

### 4.7 计数完整性不变量（硬性要求）
左侧每一项统计必须随内容变化实时、完整更新，**不得遗漏、不得对不上**：

1. **单一数据源**：所有徽标（全部 / 最新发现 / 今日速览 / 推荐深入分析 / 每个分区 / 其他）都从**同一个当前集合**（首批或增量）一次性算出，不得各算各的、不得用不同 `hours`/截断口径。
2. **全覆盖**：上述每一项都必须有计数，新增任何分区/视图时同步纳入，**不允许某项无值或漏算**。
3. **互斥求和自洽**：`全部 == 最新发现 + Σ(各分区) + 其他`（互斥划分）。「今日速览」「推荐」是同一集合上的视图（可与分区重叠），但因共用同一集合口径，**不得与「全部」相互矛盾**。
4. **随内容反应**：内容一变就重算——
   - 「更新今日新增」抓到新文章 → 下次渲染所有徽标随之变化；
   - 「标记已读」前移水位线 → 增量集合收缩 → 所有徽标随之下降；
   - 不得有缓存导致的滞后或残留旧值。
5. **可验证**：加守护测试断言上述互斥求和恒等式，并验证"更新/标记已读后计数确实变化"。

## 5. 分阶段落地

> 均为行为变更，按纪律：每阶段先快照旧行为，改后对比 + 测试锁定；不擅改已定型布局结构。

### 阶段一：增量集合 + 水位线 + 准确计数（核心）
- 引入水位线持久化 + 增量查询；主集合切到增量、去截断（留安全天花板）。
- `section_counts` 在增量全量上算（修 S5）。
- 增「标记已读」动作 + "自上次以来 N 篇"指示。
- 首次（无水位线）回退到"全部当前有效项（受 bootstrap 限定）"。
- **强制满足 §4.7 计数完整性不变量**：所有徽标同源、全覆盖、互斥求和自洽、随"更新/标记已读"实时变化。
- 验收：断言 `全部 == 最新发现 + Σ分区 + 其他`；每项徽标均有值无遗漏；模拟"更新今日新增 / 标记已读"后计数确实改变；增量为空时空态正确；首次/后续路径各自正确；quick_test / acceptance 全绿。

### 阶段二：多源公平展示 + 首次分组默认
- 增量集合内按源轮转 / 每源上限+展开（修 S2）。
- 首次默认落地"按来源分组"视图。
- 验收：构造多源数据，第一屏每源均有项；排序确定性可测。

### 阶段三（可选）：雷达内多源显示控制
- 雷达界面内"按来源筛选 / 每源开关"，与水位线正交。
- 验收：筛选只影响展示，不影响抓取与水位线。

## 6. 决策项

**已定版**：增量基准 = **每日时间锚点（默认 08:00，确定性、无需持久化）**，自动滚动，不需要用户手动「标记已读」。

待确认（建议已给，可否决）：

1. **锚点时刻 / 时区**：默认 **08:00、Asia/Shanghai(+8)**，经 `RADAR_DAILY_ANCHOR_HOUR` 等可配？
2. **主视角**：默认显示"自最近锚点以来的增量"（建议）；原 24h「今日速览」如何定位（保留为可选窗口 / 移除）？
3. **首屏默认视图**：按来源分组（建议） / 扁平时间流？
4. **多源排序**：每源上限+展开 与 轮转结合（建议） / 单一策略？
5. **安全天花板取值**：建议 2000（仅防御性，正常无感）。

## 7. 非目标

- 不动③层（摘要/洞察/报告/速览/推荐）的内部逻辑——只换②给它们的集合。
- 不动来源层 bootstrap/daily_increment 的抓取语义——只在显示层引入"首次/增量"区分。
- 不改 DB schema（水位线用运行时 JSON 持久化）。
- 不做多用户/多水位线（当前单用户，全局一个水位线）。

## 8. 建议执行顺序

先做**阶段一**（最大价值：一次性消解 S1/S3/S4/S5，并建立增量地基）→ 再**阶段二**（修 S2）→ **阶段三**按需。
